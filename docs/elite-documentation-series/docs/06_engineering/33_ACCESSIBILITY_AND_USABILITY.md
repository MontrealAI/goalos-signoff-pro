> How to make the product work for nontechnical mainstream users.

---

## Accessibility requirements

- Keyboard navigation
- Visible focus states
- Sufficient color contrast
- Labels for inputs
- Error messages tied to fields
- Plain-language status labels
- PDF receipts with readable structure

## Nontechnical usability rules

- Never show hashes as the primary explanation
- Use “receipt” before “docket”
- Use “public verification” before “on-chain anchoring”
- Hide advanced Web3 by default
- Always show next action
- Use examples in empty states

## Usability test script

Ask a new user to:

- [ ] Create a project from a template
- [ ] Invite a builder
- [ ] Find where evidence will appear
- [ ] Explain what acceptance means
- [ ] Find the receipt after acceptance
- [ ] Explain optional verification in their own words
