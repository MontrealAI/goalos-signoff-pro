PROOF-GOVERNED SWARM

Many agents become useful when the institution decides what may propagate.

The stack separates exploration from authority. Agents may propose, build, critique, and verify. Only proof that survives gates becomes memory, capability, or settlement signal.

01

Substrate layer

Compute, public-safe examples, tools, repositories, and deterministic browser scripts.

02

Agent layer

Planner, researcher, builder, critic, validator, risk sentinel, Chronicle, and human authority.

03

Coordinator layer

Evidence-state routing selects the smallest useful constellation under budget and risk.

04

Validator layer

Proof packets, replay paths, claim boundaries, and risk ledgers terminate the process.

05

Memory layer

Accepted proof becomes Chronicle memory and reusable capability; unsupported output does not.

06

Economic signal layer

The public demo emits settlement signals only. No value moves in the browser.

A

Coordination pressure

Every additional agent adds possible capability and possible waste. GoalOS treats agent count as a variable to govern, not a virtue to maximize.

B

Role-contract discipline

Each role receives scope, allowed evidence, budget, stop condition, and validator obligation. The role graph is inspectable before output is trusted.

C

Validator termination

The workflow terminates only when the required proof exists, the risk ledger is bounded, and the human acceptance boundary remains intact.

D

Experience compression

Accepted traces become routing memory. Rejected traces become warnings. The institution learns from proof, not from vibes.

E

Capability admission

A reusable capability is admitted only after replayable evidence, baseline comparison, and claim boundary review.

F

Settlement readiness

This public browser demo moves no value. It shows the control logic that future scoped settlement paths must respect.

G

Failure: swarm noise

Countermeasure: the coordinator activates only roles that reduce uncertainty or prove a claim.

H

Failure: validator drift

Countermeasure: the validator reports evidence, baseline, replay status, and risk state before any output becomes memory.

I

Failure: hidden authority

Countermeasure: human authority is explicit; no agent can self-authorize final acceptance in the public demo.

J

Failure: memory poisoning

Countermeasure: unsupported claims do not enter Chronicle memory; rejected traces become warnings, not capability.

K

Failure: reward hacking

Countermeasure: proof quality, safety, cost, uncertainty, and reviewer gates dominate any single score.

L

Failure: overclaim

Countermeasure: the page describes a browser-local synthetic demo and preserves claim boundaries while making the architecture understandable.

PROOF OF COORDINATION

The browser demonstration is intentionally synthetic, but the discipline is real.

It shows how a coordinator can route agents by evidence state, budget, risk, validator availability, stopping rule, and human acceptance boundary. The public artifact remains zero-user-data: No forms, No input, No upload, No wallet, No cookies, No analytics, No payments, and no value moved.

OPERATING RULE

Ascension is not many agents. Ascension is learned coordination under proof.

The browser demo turns that rule into a visible coordination loop: proposal, role graph, bounded work, proof packet, validation, Chronicle memory, and reusable capability candidate. Optimal effect means more verified work with less coordination entropy and fewer false acceptances.
